2. Carving Mode

Signature Carve (Selected)

What it does: Scans raw bytes for file signatures (magic bytes) without relying on filesystem metadata. Completely filesystem-agnostic — works on raw images, corrupted filesystems, unallocated space, and unknown filesystem types.

Recovers: Contiguous files only, based on header/footer signatures. No original filenames or paths — files are named by offset (e.g., jpg/f_0x1a2b3c.jpg).

Limitations: Fragmented files yield only the first fragment. No filesystem metadata means no timestamps, no filenames, no directory structure. File boundaries are estimated from signatures and size heuristics.

Best for: Heavily damaged filesystems, formatted drives, unallocated space carving, unknown filesystem types, or when filesystem metadata is unavailable.

NTFS Undelete Mode

What it does: Parses the NTFS Master File Table (MFT) to recover deleted file records. Reads file extents (data runs) directly from MFT entries and reconstructs files from their actual disk clusters.

Recovers: Original filenames, full directory paths, MACB timestamps (Modified, Accessed, Changed, Born), file attributes, and fragmented files (via MFT runlists). Files land in ntfs/<recovered_path> with original names.

Limitations: Requires intact MFT. Overwritten MFT entries or zeroed $MFT file = no recovery. Compressed/encrypted (EFS) streams not recovered in raw format. Best-effort on partially corrupted MFT.

Best for: NTFS volumes with deleted files where MFT survives. Recovers names + paths + timestamps, not just raw data. Use with BitLocker unlock if volume is encrypted.

ext2/3/4 Undelete Mode

What it does: Walks ext filesystem inodes and directory entries. Reads deleted inodes (marked free but not zeroed) and recovers filenames from directory entry slack space.

Recovers: Filenames from dirent slack, inode metadata, timestamps (mtime, ctime, atime), file permissions, and fragmented files (via extent trees or indirect blocks). Files recovered with original names where dirent slack survives.

Limitations: ext4 aggressively zeros deleted inodes and dirents — older deletions may have no name recovery. Fragmented files require intact extent trees. Journaling may have overwritten deleted metadata.

Best for: Recent deletions on ext filesystems where inodes/dirents not yet zeroed. Name recovery is hit-or-miss on ext4; ext2/ext3 fare better.

FAT12/16/32 + exFAT Undelete Mode

What it does: Scans FAT directory entries for deleted markers (0xE5). Reconstructs long filenames from VFAT LFN entries (or exFAT directory entries) and follows FAT cluster chains for file data.

Recovers: Original filenames (via VFAT LFN entries or exFAT name entries), timestamps (created, modified, accessed), file attributes, and first contiguous run of clusters. Files recovered with original names where LFN entries survive.

Limitations: Only recovers first contiguous fragment — fragmented FAT files are truncated at first gap. LFN entries often zeroed on deletion; may fall back to 8.3 short names. exFAT recovery similar but with different entry structures.

Best for: SD cards, USB drives, camera cards, embedded devices using FAT/exFAT. Quick recovery of recently deleted files with names intact.

HFS+/HFSX Undelete Mode

What it does: Walks the HFS+ catalog B-tree to find live and deleted file records. Parses fork data extents for file content and recovers metadata from catalog records.

Recovers: Original filenames, full POSIX paths, timestamps (created, modified, backed up, accessed), file/folder permissions, resource forks, and fragmented files (via extent records). Files recovered with names from catalog records.

Limitations: Clean unmounts journal away deleted catalog records — name recovery often impossible on cleanly unmounted volumes. File data still recoverable via carving mode. Catalog B-tree corruption blocks name recovery.

Best for: macOS volumes with deleted files where catalog records survive (unclean unmount, journal corruption). Data always recoverable; names only if catalog record persists.

APFS Recovery Mode

What it does: Scans APFS copy-on-write structures — file extents, directory records, and snapshot metadata. Recovers deleted files from superseded node copies that persist in unallocated space.

Recovers: Original filenames, full paths, precise timestamps (created, modified, changed, backed up, accessed), file clones, snapshots, and fragmented files (via extent lists). Exact file content via COW — no fragmentation issues.

Limitations: APFS TRIM can zero deleted blocks quickly on SSDs. Encryption (FileVault) requires unlocked volume. Snapshot-based recovery depends on snapshot age and retention.

Best for: Modern macOS volumes (10.13+). Best-in-class deleted file recovery due to COW nature — old node copies often survive long after deletion. Recovers exact file content with names and metadata.

Auto-Detect + Undelete Mode

What it does: Automatically parses MBR/GPT/APM partition tables, detects filesystem type for each partition (via magic bytes and superblock parsing), and dispatches the appropriate undelete mode (or carving fallback) to each.

Recovers: Everything the individual modes recover — filenames, paths, timestamps, fragments — across all partitions in a single run. Each partition's output lands in part<N>_<fs>/ subdirectories.

Workflow: 1) Parse partition table (MBR/GPT/APM). 2) Probe each partition for filesystem magic/superblock. 3) Run matching undelete mode (NTFS→MFT, ext4→inodes, FAT→dirents, etc.). 4) Fall back to signature carving for unrecognized filesystems.

Best for: Whole-disk images, multi-boot systems, unknown partition layouts, or when you want "everything recovered" without manual mode selection. Single command recovers all partitions.

4. Advanced Options

Offset

Start position in the source image/device. Use suffixes: 100K (kilobytes), 50M (megabytes), 2G (gigabytes). Default: 0 (start from beginning).

Use case: Skip known boot sectors, target specific partition regions, or carve only a suspicious area.

Length

Number of bytes to scan from offset. Use suffixes: 100K, 500M, 1G. Default: 0 (scan to end of source).

Use case: Limit scan to partition boundaries, specific clusters, or regions of interest identified by prior analysis.

Alignment

Only accept file headers at N-byte aligned offsets. Common values: 1 (any offset), 512 (sector-aligned), 4096 (cluster-aligned on modern filesystems).

Effect: Faster scans with fewer false positives on known filesystem images. Misses embedded files or files on non-aligned boundaries. Use 1 for raw/carving mode, higher values for filesystem-structured images.

Parallel Jobs

Number of concurrent scan processes. 0 = auto-detect (use all CPU cores). Default: 1.

Trade-off: More jobs = faster scan on multi-core systems, but higher memory usage. Use 0 or core count for large images. Single job for small images or memory-constrained systems.

Deep Validate Carves

After carving, attempt full decode of each file: JPEG/PNG structure validation, ZIP central directory check, SQLite header + page integrity, gzip stream decode, etc.

Sets confidence: verified (decode succeeded), failed (decode failed), or high (not validated). Does not modify output files.

Drop Failed Validations

When enabled with Deep Validate, discard carved files that fail structure validation. Reduces output size by removing false positives and truncated files.

Warning: May discard partially recoverable files. Use only when you need high-confidence results and can afford to lose borderline carves.

Dry Run

Scan and report findings without writing any carved files to disk. Outputs manifest JSON with offsets, sizes, and hashes, but no actual file recovery.

Use case: Quick assessment of recoverable content, estimate output size, or verify signatures match before full recovery.

Export CSV

Write findings as CSV spreadsheet in addition to JSON manifest. Columns: type, extension, offset, size, SHA-256, validated, confidence, path.

Use case: Import into Excel, Splunk, or other tools for filtering, sorting, and correlation with other evidence.

HTML Report

Generate browsable HTML report with summary statistics, file table, and image gallery (for recovered JPEG/PNG/GIF/BMP files).

Use case: Shareable reports for non-technical stakeholders, quick visual review of recovered images, case documentation.

Export Timeline

Write sorted timeline of recovered file timestamps (MACB: Modified, Accessed, Changed, Born). Output as CSV or JSONL depending on extension.

Use case: Incident timeline reconstruction, correlation with other forensic artifacts (prefetch, shellbags, event logs), bodyfile for Sleuth Kit mactime.

5. BitLocker (Optional)

BitLocker Recovery Key

48-digit Windows BitLocker recovery key (format: XXXXXX-XXXXXX-XXXXXX-XXXXXX-XXXXXX-XXXXXX-XXXXXX-XXXXXX). Unlocks BitLocker-encrypted volumes for carving/undelete.

Supported: AES-XTS-128/256 (Windows 10/11 default), AES-CBC-128/256 (older Windows), Elephant diffuser (Vista/7). Decryption is read-only and in-place — carved files are from decrypted NTFS.

BitLocker Password

User passphrase for BitLocker-protected volumes. Less common than recovery key for forensics, but supported for completeness.

Note: Passwords are case-sensitive. May fail if TPM-only protector is used (requires recovery key from AD/printed backup). Credentials are sent to the local server and passed to carvx via its environment — never on the command line.
